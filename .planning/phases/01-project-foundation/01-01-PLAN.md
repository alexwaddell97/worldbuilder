# Plan 01-01: Project Scaffolding & Design System

## Objective

Initialize the entire Next.js 15 application from scratch and establish the foundation every subsequent plan builds on. This plan wires up TypeScript strict mode, the full Tailwind CSS 4.x + shadcn/ui design system with custom CSS variable tokens, Geist Sans and Geist Mono fonts, and the Neon serverless PostgreSQL + Drizzle ORM database connection. Nothing in Wave 2 or beyond can run until this plan completes.

This is a greenfield project — the workspace contains only `.planning/` files. The Next.js project must be initialized from scratch.

## Tasks

### Task 1: Initialize Next.js 15 + TypeScript Strict Configuration

Run `npx create-next-app@latest .` in the project root (the `.` initializes in the current directory). When prompted, select: TypeScript yes, ESLint yes, App Router yes, `src/` directory yes, import alias `@/*`, Tailwind CSS no (Tailwind v4 is configured manually in Task 2).

After scaffolding, verify `tsconfig.json` includes `"strict": true` in `compilerOptions`. `create-next-app` should set this by default for TypeScript projects. The `paths` alias for `@/*` pointing to `./src/*` must also be present — confirm it is.

Create (or replace) `next.config.ts` as a TypeScript file (not `.mjs` or `.js`). Import `NextConfig` from `next` and export a minimal typed config:
- `reactStrictMode: true`
- No other options needed in Phase 1; keep it minimal

Install the `geist` npm package: `npm install geist`. This is the official Vercel package for Geist Sans and Geist Mono fonts. Do NOT use `next/font/google` for Geist — the `geist` package provides zero-layout-shift loading via pre-built CSS variables.

Update `src/app/layout.tsx` (the root layout — this is the HTML shell):
- Import `GeistSans` from `geist/font/sans` and `GeistMono` from `geist/font/mono`
- Apply `GeistSans.variable` and `GeistMono.variable` as class names on the `<html>` element alongside `lang="en"`
- The `<body>` element renders `{children}` only — no UI chrome, no hardcoded styles, just `className="antialiased"` from Tailwind
- Export a `metadata` object with `title: "Worldbuilder"` and `description: "Build richer worlds. Own your lore."`
- This layout wraps all pages (marketing, auth, app) — it must NOT contain any sidebar, header, or auth-specific UI

Verify `.gitignore` already excludes `.env.local`. If not, add it. Also confirm `node_modules/` and `.next/` are excluded.

### Task 2: Tailwind CSS 4.x + shadcn/ui + CSS Design Tokens

Install Tailwind CSS 4.x and its PostCSS plugin: `npm install tailwindcss @tailwindcss/postcss`. Do NOT install `tailwindcss@3`, `autoprefixer`, or `postcss-preset-env` — Tailwind v4 bundles its own autoprefixer and uses a dedicated PostCSS plugin.

Create `postcss.config.mjs`:
- Single plugin entry: `@tailwindcss/postcss` with no additional options
- Tailwind v4 does not require a `tailwind.config.js` — configuration is CSS-first

Run `npx shadcn@latest init` to initialize the shadcn/ui component registry. When prompted:
- Style: default
- Base color: neutral
- CSS variables: yes
- Confirm the Tailwind v4 variant if prompted — shadcn/ui as of 2025 supports Tailwind v4

This creates `components.json` and scaffolds `src/app/globals.css`. The generated `globals.css` will include `@import "tailwindcss"` (Tailwind v4's CSS-first import) and a set of CSS custom properties.

Update `src/app/globals.css` to implement the full design token system (D-14, D-15, D-17). The file must start with `@import "tailwindcss"` as the very first line. Then define all CSS custom properties under `:root` using the shadcn/ui token naming convention. Use an Apple-clean, light-mode-only neutral palette with HSL values:

- `--background`: `0 0% 99%` (near-white — primary page background)
- `--foreground`: `0 0% 9%` (near-black — primary text)
- `--card`: `0 0% 100%` (pure white for card surfaces)
- `--card-foreground`: `0 0% 9%`
- `--popover`: `0 0% 100%`
- `--popover-foreground`: `0 0% 9%`
- `--primary`: `0 0% 9%` (dark gray — primary buttons and interactive elements)
- `--primary-foreground`: `0 0% 98%` (off-white text on primary backgrounds)
- `--secondary`: `0 0% 96%` (light gray — secondary buttons)
- `--secondary-foreground`: `0 0% 9%`
- `--muted`: `0 0% 96%` (subtle background tint)
- `--muted-foreground`: `0 0% 45%` (medium gray — secondary/placeholder text)
- `--accent`: `0 0% 96%`
- `--accent-foreground`: `0 0% 9%`
- `--destructive`: `0 84% 60%` (red for destructive actions)
- `--destructive-foreground`: `0 0% 98%`
- `--border`: `0 0% 90%` (very light gray — borders and dividers)
- `--input`: `0 0% 90%` (input borders)
- `--ring`: `0 0% 9%` (focus rings)
- `--radius`: `0.5rem` (border radius)
- `--font-sans`: `var(--font-geist-sans)` — links Geist's CSS variable to the shadcn token
- `--font-mono`: `var(--font-geist-mono)` — links Geist Mono to the shadcn token

DO NOT add a `.dark {}` selector — Phase 1 is light mode only (D-17). The variable architecture is designed so a `.dark` block can be added additively in a future phase.

Set the `body` selector: `background-color: hsl(var(--background)); color: hsl(var(--foreground)); font-family: var(--font-sans);`

No hardcoded hex, rgb, or hsl values are permitted outside `globals.css`. All colors everywhere in the app must reference these CSS variables (D-15).

Update `src/app/layout.tsx` to apply the font CSS variable to `<body>`: add the `GeistSans.variable` and `GeistMono.variable` as class names (already done in Task 1 — confirm they are on `<html>`, which is the correct element for CSS variable propagation).

### Task 3: Database Connection Scaffold + Environment Template

Install the Neon serverless driver and Drizzle ORM: `npm install drizzle-orm @neondatabase/serverless`. Install Drizzle Kit as a dev dependency: `npm install -D drizzle-kit`. Do NOT install the `pg` package or `@types/pg` — the Neon serverless HTTP driver (`@neondatabase/serverless`) is used for all runtime database access instead of the standard `pg` driver. This is required for Vercel serverless + edge compatibility.

Create `src/lib/db/index.ts`:
- Import `neon` from `@neondatabase/serverless`
- Import `drizzle` from `drizzle-orm/neon-http`
- Create the Neon HTTP client: `const sql = neon(process.env.DATABASE_URL!)`
- Export the Drizzle instance: `export const db = drizzle(sql)`
- The `schema` argument will be added in Plan 01-02 once `schema.ts` exists. Leave a TODO comment: `// TODO(01-02): add schema import once src/lib/db/schema.ts is created`
- This file must NEVER be imported in Client Components — it uses `process.env.DATABASE_URL` which is a server-only secret

Create `drizzle.config.ts` in the project root:
- Import `defineConfig` from `drizzle-kit`
- `schema: './src/lib/db/schema.ts'`
- `out: './drizzle/migrations'`
- `dialect: 'postgresql'`
- `dbCredentials: { url: process.env.DATABASE_URL! }`
- This file is used by drizzle-kit CLI for schema generation and migrations only — not imported at runtime

Create `.env.local.example` in the project root. Include all environment variables the application needs, with comments explaining each:

```
# Database (Neon PostgreSQL)
# Get from: Neon dashboard → Project → Connection Details → Connection string
DATABASE_URL=postgresql://user:password@host.neon.tech/dbname?sslmode=require

# Better Auth
# Generate with: openssl rand -base64 32
BETTER_AUTH_SECRET=your-secret-here
# Full URL of the app (http://localhost:3000 in dev, https://yourapp.vercel.app in prod)
BETTER_AUTH_URL=http://localhost:3000

# Public app URL (used client-side for Better Auth client)
NEXT_PUBLIC_APP_URL=http://localhost:3000

# Google OAuth (optional — enables "Continue with Google" button)
# Get from: Google Cloud Console → APIs & Services → Credentials → OAuth 2.0 Client IDs
GOOGLE_CLIENT_ID=your-google-client-id
GOOGLE_CLIENT_SECRET=your-google-client-secret
```

Confirm `.env.local` is in `.gitignore`. The `.env.local.example` file IS committed — it documents required variables without exposing actual values.
