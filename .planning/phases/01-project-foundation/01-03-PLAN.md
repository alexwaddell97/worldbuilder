# Plan 01-03: Better Auth Configuration

## Objective

Install Better Auth 1.6.x and wire it into the application: a server-side auth instance with the Drizzle adapter, a client-side helper for use in React components, the Next.js catch-all API route that handles all auth HTTP traffic, and route-protection middleware. After this plan, the authentication infrastructure is fully in place — login/signup UI is scaffolded in Plan 01-04.

Security requirements enforced in this plan:
- httpOnly, SameSite=Lax session cookies (Better Auth default — do not override)
- CSRF protection (Better Auth built-in via the `nextCookies()` plugin)
- Email verification required before dashboard access (`requireEmailVerification: true`)
- All secrets sourced from environment variables — no hardcoded values
- Parameterized queries only — the Drizzle adapter handles this automatically

This plan runs in Wave 2 in parallel with Plan 01-02 (schema). There is no file overlap. Plan 01-03 imports `db` from `src/lib/db/index.ts` (created in Plan 01-01). The schema additions from Plan 01-02 (adding `{ schema }` to the Drizzle instance) are NOT required for Better Auth — the adapter only needs the `db` instance.

## Tasks

### Task 1: Better Auth Server + Client Configuration

Install Better Auth: `npm install better-auth`.

Create `src/lib/auth/index.ts`. This is the **server-side** Better Auth instance. This file must NEVER be imported in Client Components (`'use client'` files) — it reads server-only environment variables and configures server-side session handling.

Import `betterAuth` from `better-auth`, `drizzleAdapter` from `better-auth/adapters/drizzle`, `nextCookies` from `better-auth/next-js`, and `db` from `@/lib/db`.

Configure and export `auth`:
- `database`: `drizzleAdapter(db, { provider: 'pg' })` — this wires Better Auth's auth tables into the same Neon PostgreSQL database using Drizzle as the query layer. Better Auth generates and manages the `user`, `session`, `account`, and `verification` tables itself via its own migration.
- `secret`: `process.env.BETTER_AUTH_SECRET!` — used to sign session tokens
- `baseURL`: `process.env.BETTER_AUTH_URL!` — full URL of the application (required for OAuth callbacks and email links)
- `emailAndPassword`: `{ enabled: true, requireEmailVerification: true }` — OWASP requirement. Users cannot access protected routes until their email address is verified. Password validation: Better Auth defaults to 8+ characters minimum.
- `socialProviders`: `{ google: { clientId: process.env.GOOGLE_CLIENT_ID!, clientSecret: process.env.GOOGLE_CLIENT_SECRET! } }` — enables "Continue with Google" OAuth flow
- `plugins`: `[nextCookies()]` — required for correct cookie handling in Next.js Server Actions and RSC contexts. Without this, session cookies are not set when signing in via Server Actions.
- Do NOT configure session expiry, cookie names, or custom paths — use Better Auth's secure defaults

Export the auth instance: `export const auth = betterAuth({...})`

Export the inferred session type for use in server components and middleware: `export type Session = typeof auth.$Infer.Session`

Also export the inferred user type: `export type User = typeof auth.$Infer.Session.user`

Create `src/lib/auth/client.ts`. This is the **client-side** Better Auth instance for use in React Client Components and hooks. It must NOT import anything from `src/lib/auth/index.ts` (that is server-only).

Import `createAuthClient` from `better-auth/react`.

Create and export the auth client:
```
export const authClient = createAuthClient({
  baseURL: process.env.NEXT_PUBLIC_APP_URL,
})
```

Destructure and re-export the key hooks and methods from `authClient` for convenient imports throughout the app:
- `export const { useSession, signIn, signOut, signUp } = authClient`

`useSession` is a React hook — only use it in Client Components. `signIn`, `signOut`, and `signUp` are callable functions that work in both Client Components and Server Actions (check Better Auth docs for the correct import in each context).

Update `.env.local.example` to add `NEXT_PUBLIC_APP_URL` (needed by the auth client) with a comment explaining it is the publicly accessible URL of the app.

### Task 2: Auth API Route + Next.js Middleware

Create the directory `src/app/api/auth/[...all]/` and within it `route.ts`.

The catch-all route at `src/app/api/auth/[...all]/route.ts` is the single entry point for ALL Better Auth HTTP operations: sign in, sign up, email verification, OAuth callbacks, session refresh, sign out, etc. Better Auth's routing is handled internally.

Import `auth` from `@/lib/auth` and `toNextJsHandler` from `better-auth/next-js`.

Export the GET and POST handlers: `export const { GET, POST } = toNextJsHandler(auth)`

That is the entire contents of this file — Better Auth's handler manages all routing internally.

Create `src/middleware.ts` at the `src/` root (i.e., at `src/middleware.ts`, which Next.js detects automatically as the middleware file). This middleware enforces auth-gating on protected routes and handles the root redirect for authenticated users.

Middleware logic:
1. Read the current session using `auth.api.getSession({ headers: request.headers })` — this reads the session cookie from the incoming request
2. Determine the request pathname from `request.nextUrl.pathname`
3. **Protected paths** (require authenticated session): any path starting with `/dashboard` — if no session, redirect to `/login` with `NextResponse.redirect(new URL('/login', request.url))`
4. **Root path `/` for authenticated users**: if there IS a session and the path is exactly `/`, redirect to `/dashboard` (D-04) — `NextResponse.redirect(new URL('/dashboard', request.url))`
5. All other cases: `return NextResponse.next()`

Export a `config` object with a `matcher` that:
- Includes `/` (root, for the auth-user redirect)
- Includes `/dashboard` and all sub-paths: `/dashboard/:path*`
- Excludes Next.js internals and static assets: `'/((?!_next/static|_next/image|favicon.ico|api/auth).*)'`

Use the matcher pattern that captures both root and dashboard routes while excluding static assets and the auth API routes themselves (calling `getSession` on auth API routes causes infinite loops).

Import `NextResponse` from `next/server`.

The middleware adds ~20–40ms latency per request (one DB round-trip for session lookup). This is acceptable in Phase 1. If performance becomes a concern in later phases, cache the session in a cookie and verify locally.
