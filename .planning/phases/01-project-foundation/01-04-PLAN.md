# Plan 01-04: App Layout Shell & Route Structure

## Objective

Build the complete route architecture for Phase 1: three route groups (`(marketing)`, `(auth)`, `(app)`), functional login and signup forms wired to Better Auth, the authenticated app shell layout with a collapsible sidebar, and the empty dashboard page. After this plan completes, the full auth flow — signup → email verification → login → dashboard → logout — is testable end-to-end.

Route groups in Next.js App Router use parentheses and do NOT add URL segments. The structure is:
- `(marketing)/` — marketing pages (no auth required), has its own layout with a nav header
- `(auth)/` — login/signup pages, centered card layout
- `(app)/` — authenticated app pages, layout includes the sidebar

## Tasks

### Task 1: Route Group Layouts and Scaffolds

Create the route group directory structure and layout files. The `src/app/layout.tsx` root layout (from Plan 01-01) remains unchanged — it is the HTML/body/fonts shell only.

**Marketing group** — `src/app/(marketing)/`:

Create `src/app/(marketing)/layout.tsx`. This is a Server Component. Render:
- A `<header>` with the Worldbuilder wordmark (the text "Worldbuilder" styled as a small, clean logotype using the `--foreground` color and `--font-sans`), and right-aligned nav links: "Log in" (`/login`) and "Sign up" (`/signup`)
- A `<main>{children}</main>` for page content
- No sidebar, no auth chrome
- The header uses `sticky top-0 z-10 bg-background border-b border-border` Tailwind utilities — slim, clean, not a full navigation bar

Create `src/app/(marketing)/page.tsx`. This is the route that serves at URL `/`. This file is a **placeholder** — the full landing page content is implemented in Plan 01-05. For now, render a minimal centered page with an `<h1>Worldbuilder</h1>` and a `<p>Build richer worlds.</p>` so the route exists and the dev server does not 404 on `/`.

Add the auth redirect check at the top of the page component (Server Component — this runs server-side):
- Import `auth` from `@/lib/auth` and `headers` from `next/headers` and `redirect` from `next/navigation`
- Call `auth.api.getSession({ headers: await headers() })`
- If a session exists, call `redirect('/dashboard')` — implements D-04 (authenticated users sent to dashboard from root)

**Auth group** — `src/app/(auth)/`:

Create `src/app/(auth)/layout.tsx`. Renders a full-height centered layout for auth forms:
- Full viewport height (`min-h-screen`), flex column, centered content
- Background: `bg-muted` (the slightly off-white muted background — distinguishes auth pages from content pages)
- Children render inside a centered container of `max-w-sm w-full`
- Include a top link: "← Worldbuilder" that links back to `/` — gives users an escape route

**App group** — `src/app/(app)/`:

Create `src/app/(app)/layout.tsx`. This is the authenticated app shell. It must:
1. Verify the user is authenticated server-side: import `auth` from `@/lib/auth` and `headers` from `next/headers`. Call `await auth.api.getSession({ headers: await headers() })`. If no session, `redirect('/login')`. This is a belt-and-suspenders guard — the middleware in Plan 01-03 handles the primary redirect, but this ensures RSC rendering never proceeds without a session.
2. Import the `Sidebar` Client Component from `@/components/layout/sidebar` (created in Task 3)
3. Render a flex layout: `<div className="flex h-screen overflow-hidden">`, with `<Sidebar />` on the left and `<main className="flex-1 overflow-y-auto">{children}</main>` on the right

Create `src/app/(app)/dashboard/page.tsx`. Server Component. Reads the session to display the user's name and email:
- Call `auth.api.getSession({ headers: await headers() })` — the session object contains `session.user.name` and `session.user.email`
- Render an `<h1>Your Worlds</h1>` heading and a placeholder message: "You don't have any worlds yet. World management comes in Phase 2."
- Display the user's name and email in a subtle subheading (styled with `text-muted-foreground`)
- No world data, no CRUD — this page is a structural placeholder only

### Task 2: Login and Signup Auth Forms

Install the shadcn/ui form components needed for auth pages:
- `npx shadcn@latest add button`
- `npx shadcn@latest add input`
- `npx shadcn@latest add label`

These commands add files to `src/components/ui/`. Do NOT hand-roll these components — use the registry.

Create `src/app/(auth)/login/page.tsx` as a Client Component (`'use client'`):
- Import `signIn` from `@/lib/auth/client`
- Use `useState` for: `email`, `password`, `error` (string), `isLoading` (boolean)
- Form fields:
  - Email: `<Input type="email" id="email" placeholder="you@example.com" required />` with a `<Label>` above
  - Password: `<Input type="password" id="password" placeholder="Your password" required />` with a `<Label>` above
- Submit button: `<Button type="submit" className="w-full" disabled={isLoading}>{isLoading ? 'Signing in...' : 'Sign in'}</Button>`
- Google OAuth button: `<Button variant="outline" className="w-full" onClick={() => signIn.social({ provider: 'google' })}>Continue with Google</Button>` — place above the email form with a divider ("or")
- `onSubmit` handler: call `signIn.email({ email, password })`, handle the error response by setting `error` state, on success use `router.push('/dashboard')` (import `useRouter` from `next/navigation`)
- Display `{error && <p className="text-destructive text-sm">{error}</p>}` below the form
- Below the form: "Don't have an account? Sign up →" link to `/signup`
- Wrap the form in a `<div>` with a card-like appearance: `bg-card rounded-lg border border-border p-8 shadow-sm`

Create `src/app/(auth)/signup/page.tsx` as a Client Component (`'use client'`):
- Import `signUp` from `@/lib/auth/client`
- Use `useState` for: `name`, `email`, `password`, `error`, `success` (boolean), `isLoading`
- Form fields: Name (`text`), Email (`email`), Password (`password`) — same component pattern as login
- Google OAuth button — same as login
- `onSubmit` handler: call `signUp.email({ name, email, password })`. On success (no error returned), set `success = true` — do NOT redirect to `/dashboard`. Email verification is required (D from Plan 01-03 config: `requireEmailVerification: true`), so the user cannot log in until they verify. Show a success message instead.
- Success state: hide the form and show: `<p>Check your email! We sent a verification link to {email}. Click the link to activate your account, then sign in.</p>`
- Error state: display same as login page
- Below the form (when not in success state): "Already have an account? Sign in →" link to `/login`
- Same card wrapper style as login

### Task 3: Sidebar Component + Zustand UI Store

Install Zustand and Lucide React: `npm install zustand lucide-react`.

Create `src/stores/use-ui-store.ts`:
- Import `create` from `zustand`
- Define the store state interface: `sidebarOpen: boolean` and `toggleSidebar: () => void`
- Export `useUIStore` using `create<UIStore>((set) => ({ sidebarOpen: true, toggleSidebar: () => set((state) => ({ sidebarOpen: !state.sidebarOpen })) }))`
- This store is focused on UI state only — other stores for worlds, entities, etc. live in separate files in future phases

Create `src/components/layout/sidebar.tsx` as a Client Component (`'use client'`):
- Import `useUIStore` from `@/stores/use-ui-store`
- Import `Home` and `ChevronLeft` and `ChevronRight` icons from `lucide-react`
- Import `Link` from `next/link`

Sidebar behavior and structure:
- `const { sidebarOpen, toggleSidebar } = useUIStore()`
- Outer container: `<aside>` element with CSS transition on width:
  - Open: `w-[240px]` (per discretion noted in CONTEXT.md — standard 240px for comfortable label display)
  - Collapsed: `w-[56px]` (icon rail — wide enough for a 24px icon with 16px padding each side)
  - Transition: `transition-[width] duration-200 ease-in-out` — smooth collapse
  - Fixed height: `h-full flex flex-col`
  - Styling: `bg-background border-r border-border overflow-hidden`

- Top section: app wordmark / logo area
  - When open: "Worldbuilder" text (small, semibold)
  - When collapsed: hide text, show nothing (the icon rail starts immediately below)
  - Use conditional rendering based on `sidebarOpen`

- Nav items section (`<nav>` with `flex-1`):
  - Each nav item is a `<Link>` element with flex layout: icon on the left, label on the right
  - Open state: icon + label visible
  - Collapsed state: icon only — label hidden with `opacity-0` or `hidden` + a `title` attribute on the container for native browser tooltip
  - Phase 1 nav items:
    - Dashboard: icon `<Home size={18} />`, label "Dashboard", href `/dashboard`
  - Style active link (use `usePathname` from `next/navigation` to compare): active gets `bg-muted text-foreground font-medium`, inactive gets `text-muted-foreground hover:bg-muted hover:text-foreground`
  - Item padding: `px-3 py-2 rounded-md` for comfortable click targets

- Toggle button at the bottom:
  - `<button onClick={toggleSidebar}>` with appropriate icon: `<ChevronLeft>` when open, `<ChevronRight>` when collapsed
  - Style: `ml-auto` or positioned at the bottom of the sidebar — subtle, `text-muted-foreground hover:text-foreground`
  - Include `aria-label="Toggle sidebar"` for accessibility

All colors must use CSS variable-based Tailwind classes (`bg-background`, `text-foreground`, `border-border`, etc.) — no hardcoded color values (D-15).
