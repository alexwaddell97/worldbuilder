# Plan 01-02: Database Schema & Migrations

## Objective

Define the Phase 1 Drizzle schema for application data and generate the initial migration file. This plan implements the incremental schema strategy from D-08 and D-09: only tables actively used in Phase 1 are defined here — `worlds` and `entity_types`. The `users`, `session`, `account`, and `verification` tables are created by Better Auth's own migration (Plan 01-03) and must NOT be duplicated here.

Note on ROADMAP.md scope: The ROADMAP lists `entities`, `entity_relations`, and `reveal_rules` in Phase 1 scope, but the more specific user decision D-08 (incremental schema) and D-09 (Phase 1 tables: users via Better Auth, worlds, entity_types) override this. Defining tables for `entities` and `entity_relations` before the entity management feature exists violates the incremental principle and risks schema drift. Those tables belong to Phase 3.

This plan runs in Wave 2 in parallel with Plan 01-03 (Better Auth). There is no file overlap between the two plans.

## Tasks

### Task 1: Define Drizzle Schema

Create `src/lib/db/schema.ts`. This file defines all Phase 1 application tables using Drizzle ORM's type-safe schema API.

Import these helpers from `drizzle-orm/pg-core`: `pgTable`, `uuid`, `text`, `boolean`, `timestamp`, `unique`.

**Define the `worlds` table:**
- `id`: `uuid('id').defaultRandom().primaryKey()` — UUID primary key, generated by the database
- `slug`: `text('slug').notNull().unique()` — globally unique URL slug (e.g. `my-fantasy-world`). Unique across all worlds.
- `ownerId`: `text('owner_id').notNull()` — references the Better Auth `user` table's `id` column. Better Auth uses `text` (not `uuid`) for user IDs, so this must be `text`, not `uuid`. Do NOT add a foreign key reference at the Drizzle schema level — Better Auth manages the `user` table separately, and cross-schema foreign keys cause migration ordering issues. Enforce referential integrity at the application layer (server actions + middleware) instead.
- `name`: `text('name').notNull()`
- `description`: `text('description')` — nullable, no `.notNull()`
- `isPublic`: `boolean('is_public').default(false).notNull()`
- `createdAt`: `timestamp('created_at').defaultNow().notNull()`
- `updatedAt`: `timestamp('updated_at').defaultNow().notNull()`

**Define the `entityTypes` table:**
- `id`: `uuid('id').defaultRandom().primaryKey()`
- `worldId`: `uuid('world_id').notNull().references(() => worlds.id, { onDelete: 'cascade' })` — cascades deletion (deleting a world deletes its entity types)
- `name`: `text('name').notNull()` — display name (e.g. "Character", "Location", "Faction")
- `slug`: `text('slug').notNull()` — machine-readable slug (e.g. "character", "location"). Unique within a world, not globally.
- `icon`: `text('icon')` — nullable, stores a Lucide icon name string (e.g. "user", "map-pin")
- `isBuiltIn`: `boolean('is_built_in').default(false).notNull()` — distinguishes seeded built-in types (Character, Location, etc.) from user-created types
- `createdAt`: `timestamp('created_at').defaultNow().notNull()`
- Add a named unique constraint on `(worldId, slug)` using Drizzle's `unique()` helper — slugs must be unique within a world: `.(...).addConstraint(unique('entity_types_world_id_slug_unique').on(entityTypes.worldId, entityTypes.slug))` — or use the table-level constraints syntax depending on the Drizzle version. Confirm the correct Drizzle 0.44.x API.

Add a comment block at the top of `src/lib/db/schema.ts`:
```
// Phase 1 schema: worlds + entity_types only (D-08, D-09)
// Better Auth auth tables (user, session, account, verification) are NOT defined here.
// They are managed by Better Auth's migrate() call. See src/lib/auth/index.ts.
// Subsequent phases add their own tables (entities → Phase 3, entity_relations → Phase 5, etc.)
```

Export both tables individually and as a combined schema object:
- `export { worlds, entityTypes }`
- `export const schema = { worlds, entityTypes }`

Also export TypeScript inferred types for use in the application layer:
- `export type World = typeof worlds.$inferSelect`
- `export type NewWorld = typeof worlds.$inferInsert`
- `export type EntityType = typeof entityTypes.$inferSelect`
- `export type NewEntityType = typeof entityTypes.$inferInsert`

Update `src/lib/db/index.ts` (created in Plan 01-01) to import and wire the schema into the Drizzle instance:
- Add: `import * as schema from './schema'`
- Update the export: `export const db = drizzle(sql, { schema })`
- This enables Drizzle's relational query API (e.g., `db.query.worlds.findMany(...)`) for later phases
- Remove the TODO comment from Plan 01-01

### Task 2: Generate Initial Migration

Run `npx drizzle-kit generate` from the project root. This reads `drizzle.config.ts` and `src/lib/db/schema.ts` and generates SQL migration files in `drizzle/migrations/`. The generated file will be named `0000_initial_schema.sql` (or similar with an auto-generated suffix).

Read the generated SQL file and verify it contains:
- `CREATE TABLE "worlds" (...)` with all 8 columns and the unique constraint on `slug`
- `CREATE TABLE "entity_types" (...)` with all 7 columns, the foreign key reference to `worlds(id)` with `ON DELETE CASCADE`, and the unique constraint on `(world_id, slug)`
- NO `CREATE TABLE "users"` or any Better Auth table — confirm these are absent
- If unexpected tables appear, check that `schema.ts` does not accidentally import or re-export anything from Better Auth

The `drizzle/migrations/meta/_journal.json` file is also generated — it tracks migration history. Do not edit this file manually.

Document the migration execution process as a comment in `drizzle.config.ts`:
```
// To apply migrations to the database:
// 1. Set DATABASE_URL in .env.local (Neon connection string)
// 2. Run: npx drizzle-kit migrate
// This runs all pending migrations and records them in the __drizzle_migrations table.
// Never use drizzle-kit push in production — use migrate with tracked files only (D-10).
```

If `DATABASE_URL` is available in the environment (i.e., `.env.local` is configured with a valid Neon connection string), run `npx drizzle-kit migrate` to apply the migration and verify it completes without errors. If `DATABASE_URL` is not yet configured, note this step as a manual prerequisite for UAT and skip it for now.
